=== PROMPT 开始 ===

你是一位资深的桌面应用 UX/UI 设计师，擅长面向技术爱好者的工具型软件。请帮我设计一款 PC 桌面应用 **FH Radio Studio**，用于修改《极限竞速：地平线 6》(Forza Horizon 6) 的游戏内电台。

## 1. 背景：游戏电台机制（必读，决定所有功能）

地平线系列的电台音乐保存在两类文件里：

1. `media/audio/RadioInfo_<LANG>.xml` —— 电台元数据（歌名/作者、时间点、播放列表）
2. `media/audio/fmodbanks/R*_Tracks_CU1.assets.bank` —— 实际音频，FMOD bank 格式

关键限制：

- **不能新增曲目，只能替换**。一旦替换某电台某个槽位的歌，**该电台所有原曲都会被锁成同一首歌**。所以替换是"牺牲一个电台"的操作。
- 想保留某些原曲，可以通过 XML 把它们挂到其他电台的播放列表下。
- 每首歌需要 **6 个时间点** 才能正常播放：

| 缩写 | 含义 | XML 字段 | 单位换算 |
| --- | --- | --- | --- |
| td | 比赛开始时的播放起点（高潮起点） | TrackDrop Position | 秒 × 48000 |
| pd | 冲线动画后的播放起点（次高潮） | PostDrop Position | 秒 × 48000 |
| tl_start / tl_end | 比赛中无缝循环段的两端 | TrackLoopStart/End | 秒 × 44100 |
| pl_start / pl_end | 冲线后无缝循环段的两端 | PostLoopStart/End | 秒 × 44100 |
| 总时长 | 漫游模式切歌用 | SampleLength | 秒 × 48000 |

注意 td/pd 采样率是 48000，循环点采样率是 44100，这是游戏机制，**不能统一**。

- 必须在游戏设置里关闭电台 DJ，否则 DJ 语音会盖掉切歌时机。
- 仅 PC 端（Steam / Microsoft Store），不支持主机。

## 2. 应用要解决的三大场景

请围绕以下三个核心场景设计：

### 场景 A：替换电台中的某首歌

最高频操作。用户：

1. 选目标电台 + 目标槽位
2. 拖入自己的音乐文件
3. 应用自动响度归一化（参考原 wav 音量）
4. **AI 自动给出 td / pd / tl / pl 的候选时间点**（每个字段 top-3，带置信度和"为什么是这里"的人话解释）
5. 用户在 UI 上预听 / 微调 / 确认四个时间组
6. 一键写入游戏文件（自动备份原文件）

### 场景 B：编辑电台播放列表

不替换歌，只调整哪首歌在哪个电台里出现：

1. 显示全部电台 + 每个电台的播放列表（漫游模式 FreeRoam / 比赛模式 Event 两套独立列表）
2. 拖拽歌曲在电台之间转移
3. 删除歌曲（从某电台的列表中）
4. 复制歌曲到另一个电台（FH5 的"额外教程"功能）
5. 保存时自动同步所有语言版本的 RadioInfo_*.xml

### 场景 C：编辑已替换歌曲的 Loop / 时间点

针对已经替换过、但循环点不理想的歌：

1. 列出"已被用户替换过"的曲目
2. 选中后进入和场景 A 第 4-6 步一样的时间点编辑界面
3. 重新跑一次 AI，或者直接微调原来的值
4. 保存（只改 XML，不动 bank，速度快）

## 3. AI 候选机制（必须人在回路）

**强约束：AI 永远只给候选，不自动确认**。原因是循环点是否"无缝"是音乐审美问题，AI 会失败，用户必须能听过才点确认。

AI 的输出契约（JSON）：

```jsonc
{
  "duration_sec": 214.309,
  "candidates": {
    "td": [
      { "t": 68.72,  "score": 0.92, "why": "首个 chorus 入口" },
      { "t": 90.51,  "score": 0.61, "why": "第二段副歌" }
    ],
    "pd": [ { "t": 148.72, "score": 0.88, "why": "终段 chorus" } ],
    "tl": [
      { "start": 23.01, "end": 118.25, "score": 0.86, "bars": 32 }
    ],
    "pl": [
      { "start": 97.27, "end": 177.27, "score": 0.83, "bars": 24 }
    ]
  },
  "beats":     [/* 节拍位置数组，UI 画 beat grid 用 */],
  "downbeats": [/* 下拍位置数组，微调时磁吸用 */],
  "segments":  [/* {start, end, label: intro|verse|chorus|bridge|outro} */]
}
```

AI 失败时（任何字段无置信度 > 0.5 的候选），UI 必须明显提示"AI 信心不足，请手动指定"，但仍允许用户手点。

## 4. UI 设计硬性约束

设计时必须满足：

1. **波形 + 节拍栅格 + 段落条**三层叠加显示，所有时间点叠在同一波形上，让用户一眼看到六个点的相对位置
2. **每个时间组（td / tl / pd / pl）是独立的确认块**，三态：`待确认 → AI 已建议 → 用户已确认`。顶部状态条永远显示四组的进度
3. **试听按钮是核心交互**：
   - 单点位：从该时间起播 8 秒
   - 循环点对：从 A 前 2 秒起播 → 跳到 B → 再播 2 秒 → 重复 3 次（"试听拼接处"）
4. **微调对齐到拍**：左右按钮默认按 1 拍移动（不是固定毫秒），下拍位置带磁吸效果；按住 Shift 进入毫秒级
5. **写入游戏前的 pre-flight checklist**：弹窗列出所有六个数值（含换算后的 sample 数）和将被覆盖的文件路径，再次确认才执行（这是不可逆操作的最后一道闸）
6. **键盘快捷键**：Space 播放/暂停、数字键跳到对应段、Enter 确认当前候选、Ctrl+Z 撤销
7. **每个 AI 候选默认显示人话 `why`**（例："首个 chorus 入口"），帮非音乐背景的用户判断
8. **工程文件 .rmod.json** 保存所有状态，支持中途关闭后恢复

## 5. 技术栈倾向（可调整，给出建议即可）

我的初步倾向是 **Tauri + React + wavesurfer.js**（前端） + **Python FastAPI**（后端跑 AI 和 FMOD 调用）。如果你认为有更合适的栈（比如 Electron、Flutter、纯 Python 等），请说明取舍。

## 6. 范围外，不要设计

- 主机版（Xbox / PS）
- 全自动 zero-touch 流程
- DJ 语音替换
- 移动端
- 内置音频编辑器（裁剪、变调等都让用户在 Audacity 里做完再导入）

请按以下顺序输出，每一项尽可能详细：

1. **应用总体架构图**：模块划分（UI / AI / FMOD / XML / 工程文件），模块间数据流
2. **三个场景的完整 user flow**（场景 A、B、C 各一张流程图，标出关键决策点）

线框图请用 ASCII 画或 Mermaid，或者用 React + Tailwind 写一个可交互的 artifact 让我能直接点。

如果有任何不清楚的地方，先提问再开始设计——不要靠猜。
